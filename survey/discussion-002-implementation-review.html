A. 첫 student 스모크 5/5 = 0점 — 근본 원인 = tool-call 파싱 불일치 (length 아님)

jobs/student_smoke_20260603_130402 (Qwen3.5-4B, thinking OFF). 전부 reward 0.0. length truncation이 아니다:

  • amqp 태스크는 episode 0에서 tool call 0개로 stop. 응답(270 bytes, 잘림 아님): 
    <tool_call>
{"bash", "arguments": {"command": "cat /tmp/rabbitmq_config.conf"}}
</tool_call>
    {"bash", ...} = "name": 누락 → invalid JSON. 파서 3중 fallback이 전부 "name": 요구 → 실패 → episode 0 break → reward 0.
  • 재기동 서버에 동일 프롬프트로 재현 → 이번엔 더 줄여서: 
    <tool_call>
{"command": "cat /tmp/r.conf"}
</tool_call>
    "name"·"arguments" 둘 다 없음. _validate_tool_call 통과 불가.

즉 Qwen3.5-4B는 이 프롬프트 하에서 {"name":"bash","arguments":{"command":...}} 스키마를 신뢰성 있게 안 낸다. 서버·thinking-OFF 자체는 정상.

코드 읽고 0점 원인 판단에 동의. eval_agent.py는 XML <tool_call> JSON을 수동 파싱하고 _validate_tool_call이 name+arguments.command를 강제 → Qwen이 {"command":...}/{"bash",...}를 내면 0개 → episode 0 stop. length 아님.

D1. tool-call 경로 = native로 통일 (XML robust화는 임시 fallback만)

Step 0+는 failure/action 분포 측정 실험이라 parser artifact가 섞이면 망가짐. student도 gpt_agent.py처럼 tools=[bash, task_complete] 주고 message.tool_calls[*].function.arguments.command를 읽게. 서버는 --language-model-only 제거 + --enable-auto-tool-choice --tool-call-parser qwen3(hermes는 fallback). --reasoning-parser qwen3는 tool-call parser 대체 아님.

D2. teacher/student 일관성 = 지금은 부족

teacher native bash/task_complete vs student XML+문자열 completion → action 표현·stop 조건 다름. 둘 다 같은 BASH_TOOL/TASK_COMPLETE_TOOL schema 공유, student system prompt에서 XML 설명 제거, 완료 신호도 task_complete tool call로 통일(문자열은 legacy fallback).

D3. build_dataset.py = selection_key는 OK지만 step_log를 버림

(has_errors, -n_turns)는 arm ① dedupe로 이해 가능. 그러나 load_trial()이 messages만 저장하고 step_log를 버려서 어느 tool call이 injected error이고 어느 다음 command가 recovery인지 알기 어려움. 추천 (a): 출력에 step_log + 원본 trial path 보존. (selection_key는 유지, metadata만 살림.)

D4. 착수 순서 = tool substrate부터 고정

  1. vLLM native tool-calling smoke: 서버 플래그 수정 후 1-task로 Qwen이 bash tool call 내는지 확인.
  2. eval_agent.py를 native tool-call agent로 교체 + BASH_TOOL/TASK_COMPLETE_TOOL 공유. XML parser는 삭제/legacy 격리.
  3. trajectory log schema 통일: tool_calls, executed command, tool result, task_complete, token counts, step metadata.
  4. build_dataset.py가 step_log/source path 보존하게 고친 뒤 injected first-corrective-command extraction smoke.
  5. 그 다음에만 Step 0+ failure mining. 지금 상태로 rollout 늘리면 parser 실패를 student failure 분포로 오인할 위험 큼.

놓친 리스크: native로 바꿔도 Qwen이 arguments JSON 깨뜨릴 수 있음. smoke 기준 = reward 아니라 episode별 parsed native tool_call rate. parsed_tool_calls/model_responses, empty_tool_call_stop, invalid_arguments_json을 로그에 박을 것.

합의대로 substrate 4단계 구현·검증 완료.

서버 (D1) — parser 이름 정정: qwen3_xml

등록 parser는 hermes/qwen3_coder/qwen3_xml뿐(qwen3 없음). 실측: native tools=를 주면 chat template이 모델로 <tool_call><function=bash><parameter=command>… XML 형식을 내게 함(프롬프트로 설명만 했을 때의 hermes-JSON과 다름). hermes → tool_calls 0개(mismatch), qwen3_xml → message.tool_calls=[{bash,{command:…}}] 정상. --language-model-only는 omni 모델 LM-only 로드용 직교 플래그라 유지(제거 불필요). 결론 동일, 이름만 정정.

student (D2)
