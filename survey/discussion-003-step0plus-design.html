P0. (선결) baseline pass-rate로 task family 선정

sft_all.jsonl에 충분한 family에 native student n=3~5 → per-family pass-rate 20–70% band 5–8개(skill 다양). sampling = task 균등 아닌 failure-state 균등, 총 100–150 states.

P1. failure-state 정의·추출 ⚠

reward 0 trajectory에서 LLM-judge가 earliest-critical-error step까지 prefix(AgentDebug식). judge 출력 {critical_step_index, failure_type, recoverability, rationale}. 대안=irreversible-corruption/value-drop. 제안: earliest-critical을 primary, irreversible는 보조 플래그.

P2. progress_stage

critical_step_index/n_steps → {early<0.33, mid, late>0.66}.

P3. 7범주 라벨 룰

MEMORY 7범주(recoverable 4/partial 2/prevention 1) 그대로 + judge에 1줄 rubric·예시. recoverability는 범주에서 매핑.

P4. 3-arm hint 메커니즘 ⚠ (leak 경계)

  • no-hint: 같은 prefix 다른 seed 재샘플.
  • teacher-hint: GPT teacher가 짧은 진단만(교정 command·정답 금지). RECOVERY_PROMPT를 hint-only로.
  • oracle-hint: 다음 recovery action class(P5 키)만. 출처=known-good 복구의 첫 교정 command class. solution 아님.

leak 경계: oracle=class만, teacher=진단만, 정답 trajectory 금지. N2와 직결 → 코드/프롬프트에 명시.

P5. recovery-action-class 키

(canon_argv0, intent_class). argv0에서 path/sudo/env 제거+alias 정규화. intent_class ∈ {inspect, edit, install, run, permission, network, process, fs-mutate, other}.

P6. injected 추출 + NN (gap 지표)

sft_all.jsonl(step_log 보존)에서 intent=="error"+다음 recovery → action-class 키. student failure (family,stage,type)의 injected NN=같은 cell(희소하면 family 완화). injected_nn_same_action=class 일치. gap=일치율 <70%.

P7. 라벨 신뢰도 사전점검

20 state 2중 라벨(primary judge + 독립 2nd) + juny116 spot-check, Cohen's κ. 낮으면 rubric/키부터 수정.

P8. gate 임계(pre-registered)

gap = [recovery-action 일치율 <70%] OR [student 실패 ≥20%가 injected≈0 cell]. floor = recoverable에서 oracle success ≥40%(이하면 ②/RL pivot).

P9. 산출물 + 코드 구조

1행/state JSONL(테이블 스키마+rationale/source) + gate 스크립트. 모듈: action_class.py·injected_index.py·mine_failures.py·hints.py·recovery_eval.py·step0_table.py.

질문(우선순위)

  1. P1: earliest-critical-error primary가 맞나, irreversible-corruption primary?
  2. P4: "같은 prefix 재개" OK? oracle=action-class 제한이 N2 방어로 충분한가?
  3. P5: intent_class 룰로 충분한가 LLM 필요한가?
  4. P0/P6: cell sparsity를 family 단위로 완화하는 기준?
@codex2026-06-03 · #2

핵심 4개 결론: P1=ECE primary+irreversible 보조, P4=동의, P5=룰 primary+LLM fallback, P0/P6=단계적 backoff.

  1. P1: earliest-critical-error가 primary여야 한다(irreversible만 잡으면 너무 늦음). irreversible은 recoverability=prevention/irreversible_flag=true로. 제약: critical_step_index는 반드시 assistant bash step(observation index 찍으면 prefix 재개·action-class 추출 깨짐).
  2. P4: 같은 prefix 재개 동의. oracle=action-class만, command surface(exact argv/path/literal/patch) 절대 금지("inspect config"/"check service" 급). teacher=진단만. N=3 최소, variance 크면 5까지 config로.
  3. P5: 룰 primary + ambiguous만 LLM fallback. 매핑 제시(cat/grep→inspect, sed-i→edit, pip/apt→install, pytest/make→run, chmod→permission, curl/ssh→network, ps/kill/systemctl→process, rm/mv/cp→fs-mutate). gap키=(argv0,intent)+intent-only 병행. P7에서 fallback 비율·agreement 별도.
  4. P0/P6: 단계적 backoff. k≥5 full cell, k<5면 (family,stage)→(family,recoverability)→family 완화+match_level 기록. family에서도 k<5면 NN=unknown, 70% denominator 제외하고 injected coverage gap으로 카운트(억지 매칭 금지).

P0 추가: family 선정에 pass-rate뿐 아니라 parsed_tool_call_rate 높고 empty_tool_call_stop_rate 낮은 family만(substrate bug 재오염 방지). pilot 테이블에 세 지표 모두.

구현 gap: 모듈 순서 타당. eval_agent.py는 full conversation 저장하나 prefix-resume API 없음 → recovery eval용 "system+task user+prefix conversation 주입 후 이어 실행" entrypoint 필요.
