🎯 컨텍스트 & Thesis

Thesis (검토 대상): Recovery-training data는 teacher가 만든/주입한 failure가 아니라, 실제 target student를 rollout해서 걔가 진짜로 깨지는 지점(on-policy failure distribution)에 맞춰야 한다. Agent failure는 distributional하므로 recovery supervision은 idealized teacher가 아니라 deployed student에 align되어야 한다.

핵심 질문 3개: (1) novel한가, (2) 진짜 효과가 있나, (3) 누가 이미 했나(scoop).

🧪 제안 방법 ②½ — teacher가 복구를 시연하는 게 아님

  1. 실제 student를 rollout → 걔가 진짜 깨지는 on-policy 실패 상태 수집
  2. teacher는 복구 경로를 만들지 않고, 그 실패를 보고 힌트만 생성
  3. student가 그 힌트를 참고해 자기가 복구를 재시도
  4. 성공한 student 자신의 복구 경로를 SFT

즉 "선생은 콕 찔러주고, 답은 학생이 자기 손으로 쓴다" (STaR / rationalization 계열). hint-free vs hint-based로 "복구를 내재화했나 vs 힌트만 읽나"를 검사.

⚠️ 리스크 (방법 정정 후 우선순위)

우선순위: N2 hint-leak ≈ N1 incremental > R1(잔여) > R4 > R5. (R2 사실상 해소, R3 약화.)

N1 Incremental — 메커니즘이 STaR 계열이라 ②(From-Correction-to-Mastery)/③(SCoRe류 RL)가 인접칸 점유. reviewer는 "②½가 ② 대비 뭘 더 주냐"를 물음. 방어 = ②½ > ② 를 깔끔히 실측 (hint-only면 복구가 student 분포에 남아 더 imitable·낮은 covariate shift). 이 ②-vs-②½ 비교 자체가 핵심 contribution이 될 수 있음.
N2 Hint-leak — test엔 힌트 없음. 힌트 덕분에만 복구하면 배운 건 "복구"가 아니라 "힌트 읽기". hint-free ablation이 생사.
R1 on-policy data ≠ on-policy training (🔴→🟠 정정됨) — 복구 토큰이 teacher 게 아니라 student 자신 것이라 covariate shift 작음(STaR). 잔여: 결국 고정 데이터셋 SFT라 Wu의 "분포 맞춰도 SFT < RL"이 마진에서 물 수 있음. → RL arm을 이기려고가 아니라 gap 재려고 둠.
R4 SWE에선 레시피 이미 작동 (DAgger-LLM 2605.12913, OEC 2512.14895). wedge = terminal 도메인 + recovery-data 산출물. 정직하게 cite 필수.
R5 분포 매칭의 한계효용 — well-engineered injected와 student-mined 차이가 작을 수 있음. head-to-head에서 통계적 유의 격차 필요. ⚠ 2026-06-03 정정: literature-backed 아님 — 인용 3편(2410.15226 정반대 / 2412.02980 무관 / 2502.08661 역방향) 중 어느 것도 "Q/D > dist-matching"을 지지 안 함. R5는 순수 우리 추측 → head-to-head는 방어가 아니라 발견 실험.
R2 모티베이션 선점 = 해소/자산 — TermiGen motivation에 동의하고 그 위에서 해결책 제안. 문제를 우리가 증명 안 해도 SOTA가 해줌.
R3 Wu scoop = 약화 — "injected≠real"을 헤드라인 아닌 supporting 증거로, 그것도 Wu가 안 한 terminal에서.

📐 실험 계획 (Claude↔Codex 합의, 2026-06-03) 확정 (Codex ack, 6 rounds)

논의 전문: survey/discussion-001-first-experiment.html. 포지셔닝 — main claim: "on-policy student-mined recovery가 injected recovery를 terminal에서 이긴다(① vs ②½)". mechanism claim: "②½(hint-only self-recovery) > ②(teacher-replacement), when it works".

Step 0+ — pipeline viability + risk triage (학습 0, 제일 먼저)

Qwen3.5-4B (thinking OFF) student를 Harbor 서브셋에 rollout해 실패 상태를 채굴·계측. thesis kill 실험이 아님 — Step 1을 해도 되는지/어떤 arm을 준비할지 결정하는 전처리 gate.

  • 표본: injected(sft_all.jsonl)와 겹치는 task family 5–8개 × failure-state 15–25개 = 100–150 failure states. baseline student pass-rate 20–70% 중난이도, terminal skill 다양. sampling은 task 균등 아니라 failure-state 균등.
  • 각 state 기록(테이블 스키마): task_family, progress_stage, failure_type(7범주), recoverability(recoverable/partial/prevention), injected_nn_same_action, no_hint_success, teacher_hint_success, oracle_hint_success.
  • hint 3-arm: no-hint retry / teacher-hint / oracle(다음 recovery action class·first corrective command 수준, 정답 trajectory 아님).
  • 싼 조작화: recovery-action-class = "first corrective command 정규화 키(argv[0]+intent)" 일치. injected 쪽은 신규 rollout 불필요(sft_all.jsonl에서 추출). 본 측정 전 20개 2중 라벨로 inter-rater agreement 확인.
  • gate(비대칭): gap 검출(같은 (family,stage)서 recovery-action 일치율 <70%, 또는 student type ≥20%가 injected≈0 cell) + oracle-hint success 충분 → Step 1 go. gap 미검출 → 샘플/seed/checkpoint 확대(thesis 안 죽임). oracle로도 success 낮음 → capability ceiling, ②/RL pivot.

Step 1 — thesis go/no-go

  • ① injected-SFT vs ②½ student-mined hint-guided self-recovery SFT. ②½가 ①을 유의하게 못 이기면 핵심 thesis 실패.
  • 데이터 매칭(필수 통제): base SFT corpus 고정 + recovery slice만 swap. token budget·task coverage·failure prefix·recovery-token 비율 매칭. (a) strict-matched(causal) + (b) natural-yield(utility) 둘 다 보고.
  • eval: TerminalBench pass-rate + controlled recovery eval(hint-free). recoverable 4범주 중심, timeout/loop·reversible-corruption 별도 stratum, irreversible-destructive 제외(safety appendix).

Step 2 — isolation (Step 1 생존 시)

  • ② teacher-correction-SFT → ②½ vs ② (mechanism claim) / hint-free ablation(N2 내재화) / ③ student-mined+RL(R1 ceiling, gap 재기).

failure 7범주 층화: recoverable(wrong-cmd/API-misuse, missing-inspection, stale-assumption, verification-fail) · partial(timeout/loop, env-state-corruption=reversible vs persistent) · prevention(irreversible-destructive → 복구 불가, safety로 분리).
